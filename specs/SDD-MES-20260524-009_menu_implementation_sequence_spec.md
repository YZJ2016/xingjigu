## 2. 背景与目标

本 Spec 用于约束 005~008 的实施顺序，避免一次性大改菜单导致页面断链、文案不一致或演示数据失效。

## 3. 实施优先级

| 阶段 | 对应 Spec | 业务原因 | 主要改动类型 | 风险 |
| --- | --- | --- | --- | --- |
| P0 | 005 菜单冗余与业务边界收敛 | 先解决用户最容易误解的重叠入口 | 改名、路由、标题、说明 | 中，涉及导航和多页面文案 |
| P1 | 006 后台管理与现场终端边界治理 | 防止后台被误解为真实生产终端 | 文案、按钮、模拟回执、职责说明 | 中，涉及工位和监控页面 |
| P2 | 007 规则代码、班次日历与人员资质补齐 | 补 MES 执行底座，支撑后续准入 | 新增/扩展基础资料与系统设置页面 | 高，涉及共享数据和下游引用 |
| P3 | 008 工装校准、出货检验与 CAPA 补齐 | 补质量、设备、改善闭环 | 新增/扩展设备、质量、异常页面 | 高，涉及多个业务闭环 |

## 4. 分阶段验收门

### 4.1 P0 完成门

- 菜单无同名冲突。
- 改名后页面可跳转。
- README 描述同步。
- 重叠页面的首屏边界说明清楚。

### 4.2 P1 完成门

- 工位作业模拟动作均带“模拟”。
- 过程监控不表达真实设备控制。
- 管理驾驶舱限定为 MES 生产指标。
- 客户供应商限定为 MES 执行引用。

### 4.3 P2 完成门

- 至少能看到规则与代码、班次日历、人员资质三个业务能力。
- 开工检查、质量放行、设备点检能引用规则或资质。
- 规则发布/停用有履历。

### 4.4 P3 完成门

- 工装/校准能影响开工或检验准入。
- OQC 与 FQC/质量放行边界清楚。
- CAPA 具备 RCA、措施、验证和关闭。

## 5. 统一验证清单

每个阶段完成后必须执行：

- `node --check` 检查本阶段修改的 JS 文件。
- 静态服务打开至少 2 个代表页面。
- 验证左侧菜单跳转、当前菜单高亮、返回首页。
- 验证重置演示。
- 验证 localStorage 刷新后保留业务状态。
- 检查控制台无 JavaScript 错误。
- 检查页面文案不把后台动作表达为真实现场动作。

## 6. 回滚与兼容要求

- 菜单改名优先保留旧页面文件路径，不轻易移动 HTML 文件。
- 旧 localStorage 数据应尽量兼容；必须升级 key 时需在 README 或交付说明中注明。
- 新增页面应复用现有 CSS、JS 模块和数据模型，不新增框架。
- 一次只落地一个阶段，阶段验收通过后再进入下一阶段。

## 7. 推荐落地顺序

```text
005 菜单边界与重命名
  -> 006 后台 / 现场终端边界
  -> 007 规则代码 / 日历 / 资质
  -> 008 工装 / 校准 / OQC / CAPA
  -> README 与最终菜单业务审查
```
