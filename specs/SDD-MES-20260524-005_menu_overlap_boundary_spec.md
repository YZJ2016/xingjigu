## 2. 背景与目标

### 2.1 当前系统现状

- 当前已有菜单已覆盖 MES 正向闭环，但部分入口从业务对象看存在边界重叠。
- 已补齐 `质量检验 / 质量放行` 页面和路由，但它与 `成品检验` 仍需要清晰职责分工。
- `工序报工` 与 `工序完工`、`设备状态` 与 `设备运行`、`停机记录` 与 `停机原因 / 停机损失`、`投料确认` 与 `投料记录`、`缺料预警` 与 `缺料处理`、两个 `单据同步` 存在用户理解成本。

### 2.2 业务目标

- 让菜单入口按真实 MES 业务对象归属，而不是按页面模板或展示内容归属。
- 同一业务事实只在一个入口创建或签核，其他入口只做查询、复核、分析或下游消费。
- 用菜单名、页面标题、首屏说明和动作文案明确：哪个页面处理现场回执，哪个页面处理后台复核，哪个页面处理统计分析。

### 2.3 非目标

- 不一次性重排整个 MES 一级菜单。
- 不删除已实现页面。
- 不引入路由框架、构建工具或后端服务。
- 不把后台页面改成真实现场终端或真实外部系统。

## 3. 业务边界定义

### 3.1 需收敛的重叠入口

| 菜单 A | 菜单 B | 冗余风险 | 目标边界 |
| --- | --- | --- | --- |
| 质量检验 / 成品检验 | 质量检验 / 质量放行 | 两个页面都做 FQC 放行 | 成品检验做 FQC 检验任务、抽样、判定；质量放行做 FQC、NCR/MRB、标签、冻结、入库准入最终签核 |
| 工位作业 / 工序报工 | 完工与入库 / 工序完工 | 两个页面都做完工数量 | 工序报工接收模拟现场回执；工序完工复核数量、质量、用料、WIP 结转 |
| 设备与保养 / 设备状态 | 过程监控 / 设备运行 | 两个页面都展示设备状态 | 设备状态管 TPM 台账、准入、维修保养影响；设备运行管 PLC/SCADA 实时信号 |
| 设备与保养 / 停机记录 | 过程监控 / 停机原因 | 两个页面都登记停机 | 停机记录保留停机事实和维修关联；停机原因负责现场归因、原因代码、班组补录 |
| 过程监控 / 停机原因 | 报表与看板 / 停机损失 | 归因和分析混淆 | 停机原因用于业务归因；停机损失用于统计、下钻和改善评估 |
| 工位作业 / 投料确认 | 物料与线边库 / 投料记录 | 后台可能被理解为直接投料 | 投料确认只接收模拟扫码/PDA 回执；投料记录做查询、校验结果、批次追溯、用料差异 |
| 物料与线边库 / 缺料预警 | 异常处理 / 缺料处理 | 预警和异常处置重复 | 缺料预警识别风险；缺料处理承接已成事件的调拨、替代、催料、计划调整 |
| 完工与入库 / 单据同步 | 系统设置 / 单据同步 | 两个同名入口 | 完工与入库保留业务单据同步；系统设置改为接口补偿配置或接口对账配置 |

### 3.2 推荐命名调整

| 当前菜单 | 推荐菜单名 | 原因 |
| --- | --- | --- |
| 完工与入库 / 工序完工 | 工序完工复核 | 区分现场报工和后台完工校验 |
| 系统设置 / 单据同步 | 接口补偿配置 | 区分业务单据同步状态与接口规则配置 |
| 过程监控 / 停机原因 | 停机归因 | 强调归因，不和停机记录、停机损失混淆 |

## 4. 页面范围

### 4.1 导航入口

- 修改 `UI_Design/navigation.js` 中相关子菜单名称和路由映射。
- 同步 `UI_Design/app.js` 首页菜单映射。
- 同步所有页面使用的共享菜单渲染。
- README 记录菜单职责变化。

### 4.2 页面文案

| 页面 | 必须增加或强化的边界文案 |
| --- | --- |
| 成品检验 | “执行 FQC 检验与判定，不承担最终入库签核” |
| 质量放行 | “复核 FQC、NCR/MRB、标签、冻结和入库准入，不替代 FQC 检验或 WMS 入库” |

## 5. 数据与状态要求

### 5.1 不新增业务对象

本 Spec 优先处理菜单与职责收敛，不新增主业务数据表。现有业务事实继续由各页面和 `MES_BUSINESS_FLOW` 消费。

### 5.2 localStorage

- 菜单改名不应破坏已有业务演示数据。
- 如果页面 `STORAGE_KEY` 与页面名称无关，不升级版本。
- 如页面状态中持久化了旧菜单名，需要兼容旧值并在保存时写入新值。

## 6. 交互规格

| 动作 | 前置条件 | 成功结果 | 失败结果 |
| --- | --- | --- | --- |
| 点击改名后的菜单 | 页面存在路由 | 进入原业务页面，标题显示新业务定位 | Toast 或无效路由不得出现 |
| 从首页搜索旧菜单名 | 用户搜索旧称 | 可通过别名命中或在文案中看到新名称 | 不应搜索不到核心业务入口 |
| 重置演示 | 已产生本页状态 | 保留新菜单名称和默认展开策略 | 不恢复旧名称 |

## 7. 验收标准

- 所有改名菜单可从首页和子页面侧边栏进入。
- `质量放行` 与 `成品检验` 页面首屏说明不再重复表达为同一个 FQC 动作。
- `工序报工` 与 `工序完工复核` 的页面按钮和说明分别体现“模拟现场回执”和“后台复核”。
- 两个 `单据同步` 不再同名。
- 浏览器控制台无 JavaScript 错误。
- 不引入新依赖，不新增真实后端。

## 8. 实施顺序

1. 改名并补齐导航映射。
2. 同步首页搜索、常用入口、README。
3. 修改相关页面标题、副标题、按钮文案。
4. 检查 localStorage 旧值兼容。
5. 静态服务逐页验证菜单跳转。
