# AGENTS.MD

## 项目定位

这是星技谷 MES 前端业务原型项目，用于沉淀真实 MES 生产业务规则、业务蓝图、正向业务闭环说明和工作台/业务页面。所有页面、交互和数据组织必须以实际 MES 生产业务为准，而不是以演示效果、展示包装或视觉陈列为修改目标。当前项目不依赖构建工具，核心页面可以直接通过浏览器打开。

后续所有代理协作都应优先保持项目的静态、轻量、可直接预览特性。除非任务明确要求，不引入框架、包管理器、打包器或真实后端服务。虽然当前是纯前端项目，但 JavaScript 与 localStorage 必须承担“前端内置后端”的职责：用领域数据模型、业务状态机、校验规则、模拟接口、持久化和审计记录组织业务，而不是只为页面渲染临时拼接数据。


## 协作原则

1. 始终以真实 MES 和真实工厂生产业务为设计与修改依据。
   - 当前 UI 原型以 MES 后台管理系统为主视角。后台页面应面向计划员、车间主任、班组长、工艺员、质量员、设备员、物料员、仓储员、系统管理员等管理和协同角色。
   - 所有改动必须先明确对应的真实生产业务目标，例如计划下达、工序派工、工位开工准入、投料防错、过程检验、设备点检、异常处置、完工报工、批次追溯或班次交接；不得以“更像演示”“更好看”“更丰富”为首要目标。
   - 页面、字段、状态、按钮和交互必须能解释其在真实 MES 闭环中的来源、去向、责任岗位、业务单据、校验规则和记录结果。
   - 页面设计必须先回答真实现场中“谁在什么岗位、使用什么设备或终端、处理哪张单、执行哪个动作、产生什么业务记录”。
   - 不把页面设计成脱离现场的泛后台管理系统。后台页面必须体现工位终端、设备 HMI、PDA、扫码枪、工牌/NFC、PLC/设备信号、ERP/PLM/WMS/QMS 等真实来源产生的数据、状态、校验结果和异常。
   - 后台页面不应替代现场终端执行真实生产动作。扫码开工、投料、报工、检验、设备点检、工位签收等现场动作，应在后台体现为配置下发、执行监控、状态回执、异常处置、追溯查询、补偿记录或审核放行。
   - 新增页面必须体现真实 MES 的业务闭环：状态来源、操作校验、责任人、时间戳、单据或批次关联、异常处理、交接或追溯结果。
   - 页面文案应使用工厂现场语言和 MES 业务语言，例如“工位绑定、开工准入、投料防错、工序报工、首件确认、班次交接”，避免把现场身份确认表达成普通网页登录。
   - 强制区分后台管理动作、真实生产动作和静态演示中的模拟外部操作。凡是在页面中用按钮、输入框或控件模拟设备 HMI、扫码枪、PDA、工牌刷卡、NFC、设备账号、PLC/设备信号、外部系统回传等外部动作时，控件文案、区域标题或占位提示必须明确出现“模拟”字样，避免被误解为后台直接执行真实生产动作。
   - 如果后台页面展示 SOP、工艺参数、图纸、检验规范等资料，应优先体现来源系统、版本、生效范围、审批状态、下发状态、现场签收和追溯引用；除非任务明确要求，不把 MES 后台设计成 PLM、QMS、设备程序库或文档编辑器。
   - 如当前纯前端交互无法完整覆盖真实业务链路，必须在页面和 JS 数据结构中保留可继续扩展的位置，例如履历、明细、交接记录、异常原因、追溯关联、接口回执、审批记录等。

2. 先读业务文档，再改 UI。
   - 页面内容应与根目录的 MES 流程说明、闭环示例和蓝图资料保持一致。
   - 新增模块、字段、流程节点时，需要确认其在业务链路中的位置。

3. 保持静态项目边界。
   - 默认只编辑 HTML、CSS、JavaScript 和 Markdown。
   - 不新增运行时依赖，不引入构建步骤。
   - 如确需新增依赖，必须先说明原因、收益和替代方案。
   - 即使当前是静态前端，也必须按正规前端工程方式组织代码；不要把新功能持续堆在单个 HTML、CSS 或 JS 文件中。
   - 不新增真实后端时，应在前端用 JS 模块和 localStorage 清晰表达后端职责边界，包括数据表/集合、业务服务、状态流转、校验、查询、写入、重置和审计留痕。

4. 遵循现有文件分工。
   - 当前首页 `index.html` 只放页面结构和语义容器。
   - 当前首页 `styles.css` 只放视觉、布局、响应式和状态样式。
   - 当前首页 `app.js` 管理前端内置业务数据、localStorage、事件绑定和 DOM 渲染。
   - `README.md` 记录面向使用者的打开方式和页面范围。
   - 当功能复杂度继续上升时，应优先新增独立页面或拆分为 `styles/`、`js/`、`features/`、`data/` 等模块目录；入口 HTML 只负责装配资源和提供页面挂载结构。
   - 新增一级菜单或完整子菜单功能时，应建立对应的独立页面或 feature 模块，例如 `orders/index.html` 或 `features/orders/`，不要把完整业务功能直接塞回首页文件。

5. 控制改动范围。
   - 只修改和任务直接相关的文件。
   - 不做无关重构、格式化或文案大改。
   - 不覆盖用户已有修改；遇到不相关的脏工作区内容时保留原状。

## 开发规范

### 产品与信息架构

- UI 应按真实 MES 使用场景组织，而不是直接照搬蓝图中的技术模块划分。
- 每个页面都应根据自身业务目标、使用角色、信息密度和操作路径选择最适合的页面布局，并在画面层级上突出当前页面最重要的业务重点；不要因为项目中已有某种布局，就默认所有页面都复用同一套布局。
- 左侧菜单使用业务语言，例如“订单与计划”“派工与生产任务”“工位作业”“物料与线边库”“质量检验”“设备与保养”“异常处理”等。
- 避免在业务菜单中出现偏技术实现的词，例如“状态机”“死信”“幂等”“接口可靠性”“MTBF/MTTR”等；这些能力可以存在于系统设置、说明文档或后续技术页中。
- 左侧菜单保留彩色标识块，但不使用 `01/02` 这类顺序编号；使用业务含义短标识，例如“首、计、任、位、料、质、设、异”等。
- 菜单默认收起。用户展开/收起状态应通过 `state.openMenus` 和 localStorage 保存。

- 抽屉、弹窗、Toast 等依赖 `hidden` 属性显示隐藏时，CSS 必须保留 `[hidden] { display: none !important; }`，避免被组件 `display` 样式覆盖。
- 抽屉、弹窗、悬浮窗、右侧详情面板等会遮挡或挤占主画面的组件，必须提供明确的关闭按钮；关闭后需要有可发现的重新打开入口。若关闭/展开属于演示状态，应写入 localStorage 并支持重置演示恢复。

### 前端内置后端与 localStorage

- 当前没有真实后端时，必须用 JavaScript + localStorage 组织出可追溯的业务后端形态，而不是只保存页面控件状态。
- JS 数据应按真实 MES 领域对象建模，例如订单、工单、工序任务、工位、设备、人员、物料批次、检验记录、异常单、审批/放行记录、接口回执、操作日志等。
- 业务写入必须经过明确的服务函数或业务动作函数，集中处理状态校验、字段补齐、时间戳、责任人、来源终端、关联单据和审计记录；避免在 DOM 事件里直接零散改对象。
- localStorage 应作为前端内置持久化层，保存业务数据、业务状态、查询筛选和必要的页面展开状态，支持刷新后继续按真实业务链路查看。
- 当前页面应保存：
  - 订单数据
  - 现场任务
  - 风险事项
  - 待办事项
  - 当前筛选状态、当前订单、订单池展开状态、菜单展开状态
- 顶部保留“重置演示”入口，用于清空 localStorage 并恢复初始业务种子数据。
- `STORAGE_KEY` 变更要有意识：当新增初始示例数据且希望用户自动看到新数据时，可以升级版本号；如果要保留用户演示过程数据，不要随意升级。
- 新增业务数据时，应同步考虑真实关联关系、状态流转、查询、KPI、产线负荷、异常处理、追溯和重置场景。

### HTML

- 使用语义化结构，保留必要的 `aria-label`、按钮类型和表单标签。
- 页面入口继续使用 `UI_Design/index.html`。
- 避免在 HTML 中内联大段样式或复杂脚本。

### CSS

- 沿用现有 CSS 变量、面板、按钮、网格和状态类命名风格。
- 新增样式优先复用现有色板：`--blue`、`--green`、`--orange`、`--red`、`--teal`、`--indigo` 等。
- 避免让页面变成单一色系；保持生产运营工具应有的克制、清晰和可扫描性。
- 关注固定宽度、最小宽度、滚动区域和文本溢出，避免控件和文字重叠。
- 主内容区采用左右独立纵向列，避免 CSS Grid 行高互相牵连造成大面积空白。
- 订单池需要支持折叠/展开。默认只展示少量重点订单，展开后展示全部订单；超过可视区域时应通过页面滚动或抽屉滚动处理。

### JavaScript

- 使用原生 JavaScript，不引入框架。
- 业务种子数据优先放在 JS 顶部或独立数据模块中的结构化数组/对象里，字段命名应贴近 MES 业务语义。
- 复杂页面应拆分数据、业务服务、渲染和事件绑定，不把“模拟后端”的数据读写、业务校验和 DOM 拼接混在同一个大函数里。
- 对外部系统、设备、PDA、扫码枪、HMI、PLC 等输入的模拟，必须在 JS 中表现为带来源、时间和回执的事件或接口结果，并写入 localStorage 的业务记录。
- 新交互需要同步考虑：
  - 初始状态
  - `localStorage` 保存和重置
  - 业务校验和失败原因
  - 操作日志或状态履历
  - 渲染函数
  - 事件绑定
  - 空状态或异常状态
- 修改 localStorage key 时要评估是否会破坏已有演示数据。

### 文案

- 使用中文，面向 MES 生产、计划、质量、设备、仓储等业务角色。
- 避免营销化表达，优先使用业务动作、状态、单据、责任人和闭环结果。
- 示例数据可以虚拟，但要符合制造现场常识。

## 运行与验证

本项目可以直接打开：

```text
UI_Design/index.html
```

也可以在项目根目录或 `UI_Design` 目录启动任意静态文件服务。示例：

```bash
python3 -m http.server 8765
```

验证重点：

- 首页能正常加载，无控制台 JavaScript 错误。
- 菜单、搜索、筛选、订单切换、KPI、抽屉/弹窗、待办勾选、重置演示等交互可用。
- 刷新页面后，演示状态能通过 localStorage 保留。
- 点击“重置演示”后，数据回到初始状态。
- 页面主要区域没有明显遮挡、溢出或不可读文本。

## 常见任务流程

### 新增业务模块

1. 在业务文档中确认模块所属流程。
2. 更新 `app.js` 中的模块、数据或状态。
3. 如需要新增展示区，再更新 `index.html`。
4. 在 `styles.css` 中补齐布局和状态样式。
5. 更新 `UI_Design/README.md` 的页面范围说明。
6. 手动打开页面验证交互和视觉。
7. 功能与功能之间要考虑业务连贯性

### 新增业务数据

1. 优先使用结构化对象，不把业务字段散落在渲染字符串里。
2. 确认订单、产线、工序、风险、待办之间的引用关系一致。
3. 补齐状态来源、责任岗位、时间戳、关联单据、异常原因、追溯链路和操作日志等真实 MES 所需字段。
4. 覆盖筛选、搜索、统计、业务校验、刷新持久化和重置场景。

### 调整视觉布局

1. 先确认目标视口和业务使用场景。
2. 优先微调现有布局规则。
3. 检查侧边栏、顶部操作区、主内容网格和面板内部密度。
4. 避免新增装饰性元素影响信息扫描。
5. 如果有抽屉、弹窗、悬浮窗或右侧详情面板，需要提供关闭功能和重新打开入口。

## 交付清单

每次完成较大改动后，代理应在回复中说明：

- 修改了哪些文件。
- 完成了哪些业务或交互目标。
- 做了哪些验证。
- 是否存在未验证项或后续风险。

## 禁止事项

- 不要提交生成目录、缓存目录或临时截图。
- 不要无理由引入 npm、Vite、React、Vue 等工程化依赖。
- 不要把根目录业务文档中的内容大段复制进页面，页面应提炼为可操作工作台。
- 不要移除现有业务说明文件。
- 不要修改与当前任务无关的大段格式。
