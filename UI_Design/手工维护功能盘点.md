# MES 页面手工维护功能盘点

## 说明

本文按现有 `UI_Design` 页面逐项筛选后台用户可以手工维护的内容，用于后续 UI 原型补齐“新增、修改、删除/停用、审批、导入、作废、恢复、留痕”等基础管理能力。

判定原则：

- 后台可维护：由计划员、工艺员、质量员、设备员、仓储员、系统管理员等在 MES 后台配置、申请、审批、补录或关闭的业务对象。
- 后台不可直接执行：扫码开工、投料、设备启动、PLC 采集、PDA 签收、工牌/NFC 登录、WMS 上架、ERP 入账等真实现场或外部系统动作，后台只能查看回执、配置规则、模拟演示、补偿、审核或异常处置。
- 删除优先表达为停用、作废、撤回、冻结、关闭、归档；已被生产、质量、入库、追溯引用的数据不应物理删除。
- 所有手工维护动作都应写入 localStorage 演示状态，并在页面体现责任人、时间戳、关联单据、前后值摘要和操作记录。

## 改造优先级

1. **P0 主数据与系统治理**：基础资料、系统设置。它们天然需要新增、编辑、停用、版本发布、审批和影响评估，也是其他页面的业务底座。
2. **P1 计划与执行管理单据**：订单与计划、派工与生产任务、异常处理、设备计划、物料申请、质量任务、完工入库。它们需要用户发起、调整、审批、关闭和补录。
3. **P2 现场回执与监控类页面**：工位作业、过程监控、追溯、报表。它们主要是查询、复核、模拟回执、异常转单、导出或发布，不应设计成后台直接替现场执行。

## 页面布局业务重点评审与执行记录

### 评审结论

当前多数页面的业务文案、数据来源、责任人、时间戳和模拟外部回执表达基本符合 MES 现场边界，但 `物料与线边库`、`质量检验`、`设备与保养`、`过程监控`、`异常处理`、`完工与入库`、`追溯查询` 等模块仍大量复用 `指标 + 表格 + 卡片 + 模拟回执 + 右侧详情/履历` 的同一工作台骨架。后续改造目标不是取消共用骨架，而是在首屏增加每类业务自己的主视觉和操作重点，让用户一眼知道当前页面处理的是产量、设备、参数、报警、停线、缺料、质量、维修、复盘、入库校验还是证据链查询。

### 布局改造优先级

1. **第一轮：过程监控、异常处理**。这两个模块最容易被识别为套模板，应优先增加页面专属重点区：过程监控突出产量节拍、设备状态、参数规格、报警等级、停机归因、SPC 趋势和电子看板发布；异常处理突出异常来源、SLA、分派升级、停线授权、缺料处置、质量围堵、设备维修和 RCA/CAPA。
2. **第二轮：物料与线边库、设备与保养、完工与入库**。物料页应转为需求缺口矩阵、配送路径、线边库位状态、投料防错、退料核销和缺料影响分析；设备页应转为设备状态矩阵、点检/保养日历、维修响应闭环、停机归因和 OEE 三分项；入库页应突出完工校验门、包装层级、WMS 回执、冻结/解冻和 ERP/WMS 对账补偿。
3. **第三轮：质量检验、追溯查询、报表与看板**。质量页应突出检验放行门、巡检路线、SPC/NCR/MRB；追溯页应突出证据链、影响范围、查询快照和客户报告；报表页应降低明细表首屏权重，强化发布口径、指标下钻和经营驾驶舱。

### 第一轮执行计划

范围：

- `过程监控` 全组页面。
- `异常处理` 全组页面。

目标：

- 在保留现有静态项目、localStorage、模拟回执、右侧详情和表格查询能力的前提下，给每个子页面增加一个首屏业务重点区。
- `过程监控` 每个子页面应形成不同的信息布局：产量达成、设备状态矩阵、参数规格门、报警分级流、停机归因条、SPC 趋势卡、电子看板发布屏。
- `异常处理` 每个子页面应形成不同的信息布局：异常来源分级、SLA 队列、停线授权门、缺料处置路径、质量围堵/MRB、设备维修响应、RCA/CAPA 复盘。
- 新增区域继续明确“模拟”外部回执边界，不把后台按钮表达成真实生产、设备或仓储动作。

验收点：

- 页面首屏不再只有指标与表格，能直接看到该子页面的业务主对象和闭环重点。
- 点击重点区对象可切换右侧详情，状态、履历和模拟记录继续可用。
- 桌面宽度下不出现明显文字遮挡；窄屏下重点区能单列展示。
- 不引入框架、构建工具或运行时依赖。

## 业务闭环分轮实施计划

目标：所有页面按钮都按 MES 正向业务闭环推进统一业务状态，而不是只写入某个页面自己的临时记录。静态演示仍使用浏览器持久化保存演示状态，但语义上统一为 `MES_BUSINESS_FLOW` 业务流程状态，订单、评审、排程、派工、物料、质量、设备、异常、入库、追溯和报表都从同一条业务链路读写。

### 第一轮：计划执行前置闭环

范围：

- 订单与计划
- 基础资料
- 派工与生产任务
- 物料与线边库

目标：

- 打通 `ERP/MES 订单 -> 主数据校验 -> 订单评审 -> APS 排程 -> 齐套检查 -> 派工准备 -> 物料需求/领料/配送`。
- 生产订单新增、复制、提交评审、评审通过、发布排程、确认齐套等动作写入统一业务流程状态。
- 基础资料发布、撤回、停用等动作能影响被订单引用的主数据校验结论。
- 派工下达、任务接收、开工检查、物料领料/配送/线边库存等动作消费前序业务事实，不再自造孤立单据。

状态：第一轮后续已完成。已建立 `MES_BUSINESS_FLOW` 统一业务流，订单复制、新增、评审、排程、齐套、基础资料变更、派工下达和物料回执都会写入同一份业务状态；派工单和物料页面已优先合并统一业务流派生出的工序任务与用料需求，新增/复制订单后可继续进入派工与物料链路。页面自身筛选、详情开合等仍保留本页 `localStorage`，但业务结果不再只停留在单独页面。

### 第二轮：现场执行与过程采集闭环

范围：

- 条码与标签
- 工位作业
- 过程监控

目标：

- 打通 `派工下达 -> 批次/SN/标签生成 -> 模拟扫码开工 -> 投料防错 -> 过程参数/设备信号回执`。
- 现场动作保持为“模拟外部回执”或“后台复核/拦截/补偿”，不把后台按钮设计成真实开工、投料或设备控制。

状态：第二轮已完成主干闭环。条码与标签、扫码开工、投料确认、过程记录、过程监控已接入 `MES_BUSINESS_FLOW`：批次/SN/标签下发写入条码事件，模拟扫码开工和设备启动信号写入工位事件，投料确认/拦截联动物料与工位事件，过程参数/设备/报警/停机/看板模拟回执写入过程事件；这些事件同步进入订单追溯事件，为第三轮质量、异常、设备闭环提供统一事实来源。

补充状态：第二轮非主干已补齐。员工登录、工艺指导、工序报工、交接班已接入统一工位事件；标签作废/补打拦截等非主动作进入条码审计事件；过程监控人工复核进入过程事件。至此第二轮范围内的现场执行、标签审计、工位签收、报工补偿、交接移交和过程复核都具备统一业务事实来源，后续质量/异常/设备模块可以消费这些事件继续闭环。

- 质量检验
- 异常处理

- 打通首件、巡检、过程检验、FQC、不良、返工、设备报警、维修、停线、缺料、质量问题的统一异常闭环。
- 质量和设备结论反向影响开工准入、报工、排程调整、库存冻结和追溯。

状态：第三轮已完成主干闭环。质量检验、异常处理、设备与保养已接入 `MES_BUSINESS_FLOW`：首件、巡检、过程检验、FQC、不良、返工等质量动作写入质量事件；异常派单、临时措施、停线/缺料/质量/设备异常处置写入异常事件；设备状态、点检、保养、维修、停机、备件和 OEE 复核写入设备事件。第三轮页面会消费第二轮产生的工位、投料、过程采集、设备信号和条码事实，并将质量/异常/设备结论回写到订单风险、异常事实、派工设备状态和追溯事件，为第四轮完工、入库、追溯和报表消费提供统一依据。

### 第四轮：完工、入库、追溯和经营结果闭环

范围：

- 完工与入库
- 追溯查询
- 报表与看板

目标：

状态：第四轮已完成主干闭环。完工与入库、追溯查询、报表与看板已接入 `MES_BUSINESS_FLOW`：工序完工、完工确认、包装、成品入库、库存冻结、退料入库、ERP/WMS 单据同步等动作写入完工入库事件；追溯查询从完工入库事件派生产品、批次、物料、生产、检验、设备和客户报告证据链，并将查询快照、影响范围复核写回统一追溯事件；报表与看板从完工入库和追溯事实派生日报、良率、交付、OEE、停机、损耗和驾驶舱经营口径，并将模拟刷新、复核结论写入统一报表事件。第四轮页面仍保留本页筛选、详情开合和模拟记录 localStorage，但业务结果不再只停留在单独页面。

## 原手工维护分轮计划

### 第 1 轮：系统设置基础维护闭环

目标：先补齐最典型的后台维护体验，解决“人员账号没有新增、修改、离职停用”等问题，并沉淀可复用的维护表单、行操作、审计记录交互。

范围：

- 系统设置 / 人员账号
- 系统设置 / 角色权限
- 系统设置 / 审批设置
- 系统设置 / 消息提醒

主要动作：

- 增加 `新增`、`编辑`、`复制`、`停用/启用`、`锁定/解锁`、`提交审批`、`查看履历`。
- 人员账号重点补：新增人员、编辑岗位班组、离职停用、绑定工牌/NFC/PDA、重置令牌、权限复核。
- 角色权限重点补：新增角色、授权用户、数据范围、按钮权限、复制角色、停用角色。
- 审批设置重点补：新增流程、编辑节点、复制模板、发布/停用版本。
- 消息提醒重点补：新增规则、编辑模板、启停规则、模拟测试发送。

验收点：

- 表单提交后写入 localStorage，刷新后保留。
- 停用、锁定、发布等动作进入操作记录列表。
- 危险动作有二次确认。
- 页面不再只有内置数据展示，至少能完整完成一条新增-编辑-停用链路。

### 第 2 轮：基础资料主数据维护闭环

目标：补齐 MES 业务底座的主数据维护能力，并建立版本、审批、生效范围、下游影响评估的统一模式。

范围：

- 基础资料 / 产品资料
- 基础资料 / 物料资料
- 基础资料 / BOM 清单
- 基础资料 / 工艺路线
- 基础资料 / 工序工位
- 基础资料 / 产线车间
- 基础资料 / 客户供应商

主要动作：

- 增加主数据 `新增`、`编辑`、`复制版本`、`停用`、`导入`、`提交审批`、`发布到执行`、`撤回未生效版本`。
- BOM 和工艺路线要体现明细维护入口，但不做复杂 PLM 编辑器。
- 工序工位、产线车间要突出设备、终端、班组、班次、开工准入规则维护。
- 客户供应商要突出客户标签要求、供应商质量等级、追溯口径维护。

验收点：

- 新增主数据可参与筛选、详情和卡片统计。
- 已发布与待审批状态区分清楚。
- 停用旧版不影响已被订单引用的数据，只提示下游影响。
- 页面明确说明外部 ERP/PLM/QMS/WMS 同步是“模拟”，MES 只做执行快照和审批记录。

### 第 3 轮：订单与计划维护闭环

状态：已完成。生产订单、订单评审、生产排程、产能负荷、交期预警、计划调整、齐套检查已接入统一计划维护面板，维护记录写入 `localStorage.planMaintenanceLogs` 并同步订单状态与最近记录；生产订单删除已按引用状态区分未引用删除与已引用作废。

目标：把生产订单到排程、齐套、交期和调整的计划员操作补完整，形成计划部门真实可维护工作台。

范围：

- 订单与计划 / 生产订单
- 订单与计划 / 订单评审
- 订单与计划 / 生产排程
- 订单与计划 / 产能负荷
- 订单与计划 / 交期预警
- 订单与计划 / 计划调整
- 订单与计划 / 齐套检查

主要动作：

- 生产订单保留并完善已有新增、编辑、删除能力，补作废/暂停、复制、导入、提交评审、影响校验。
- 订单评审补资料问题、会签意见、退回原因和评审履历维护。
- 排程补排程草案、工序时间编辑、拆批、锁定、发布、撤回。
- 产能负荷补加班窗口、停线窗口、资源约束、均衡建议。
- 交期预警补风险新建、原因编辑、责任分派、升级关闭。
- 计划调整补申请、审批、生效、回退链路。
- 齐套检查补替代料、拆批建议、手工复核和开工拦截。

验收点：

- 计划类单据之间能通过状态和快捷入口串联。
- 已下达或已引用数据不能物理删除，只能作废、暂停、撤回或调整。
- 不出现后台直接启动产线、设备或修改 ERP 原始订单的表达。

### 第 4 轮：派工、物料、质量、设备业务单据维护

目标：补齐车间执行协同中的管理类单据，让用户能发起、调整、审批、关闭，而不是只看状态。

范围：

- 派工与生产任务全组
- 物料与线边库全组
- 质量检验全组
- 设备与保养全组

主要动作：

- 物料：新建领料/退料/调拨建议，编辑数量批次，提交审批，重推 WMS，关闭差异。
- 质量：新建检验任务、不良记录、返工评审，录入/复核结论，放行/隔离/驳回。
- 设备：新增设备台账、点检/保养计划、维修工单、备件申请，补录停机归因，关闭验收。

验收点：

- 每类单据至少有一条完整新增-处理-关闭演示链路。
- 现场动作保持为回执、模拟或复核，不设计成后台直接替工位执行。
- 关键操作都有责任人、时间戳、关联单据和操作记录。

### 第 5 轮：异常处理、完工入库、条码标签维护

目标：补齐异常、条码、入库这些跨部门闭环，重点处理审批、作废、补打、冻结、补偿等强审计动作。

范围：

- 异常处理全组
- 完工与入库全组
- 条码与标签全组

主要动作：

- 异常：新建异常、派单、转派、升级、恢复验证、关闭、复盘 CAPA。
- 完工入库：完工复核、入库指令、WMS 重推、冻结/解冻、退料入库、接口补偿。
- 条码标签：批次/SN 生成、标签任务、打印重试、作废、补打申请、审批、新旧标签关系。

验收点：

- 补打、冻结、解冻、接口补偿等危险动作必须有审批或二次确认。
- 条码和追溯相关数据不允许无痕删除。
- 入库和同步页面明确不直接修改 WMS/ERP 账务。

### 第 6 轮：监控、追溯、报表的复核与发布能力

目标：给查询和看板类页面补齐业务上合理的手工动作，但不把它们改成数据维护后台。

范围：

- 工位作业全组
- 过程监控全组
- 追溯查询全组
- 报表与看板全组
- 首页工作台入口联动

主要动作：

- 工位作业：补录原因、复核异常、解除拦截申请、查看履历、转异常。
- 过程监控：报警派单、参数复判、停机原因复核、趋势批注、看板发布配置。
- 追溯查询：保存查询、生成快照、导出报告、转冻结/异常/客户报告。
- 报表：生成快照、复核、发布、撤回、导出、生成改善项。
- 首页：把新增异常、风险关闭、常用入口配置与具体页面串联。

验收点：

- 监控、追溯、报表只维护复核结论、快照、批注、导出、发布状态，不改原始事实。
- 所有模拟外部动作继续明确出现“模拟”字样。
- 首页能作为常用维护入口，不重复承载完整业务页面。

## 每轮交付方式

每轮建议按以下顺序执行：

1. 先选定本轮页面的共用交互模型：表单、详情、履历、危险确认、Toast、localStorage 字段。
2. 先做 1 个代表页面作为样板，例如第 1 轮先做“人员账号”。
3. 样板验收后，再扩展同组页面，减少重复返工。
4. 每轮结束更新 README 和本盘点文档中的完成状态。
5. 每轮至少验证桌面宽度页面加载、主要新增/编辑/停用动作、刷新后 localStorage 保留、重置演示恢复。

| 操作记录 | 审计查询条件、归档任务、导出申请 | 查询、筛选、导出申请、归档、标记稽核样本、查看前后值 | 不允许编辑或删除原始审计记录 |
| 数据备份 | 备份策略、归档策略、恢复演练计划 | 新增策略、编辑 RPO/RTO、启停策略、发起演练、确认结果、告警配置 | 不在演示页面直接访问生产数据库 |

## 通用 UI 补齐建议

每个可维护页面建议统一补齐以下控件和状态：

- 顶部主按钮：`新增` / `新建申请` / `生成任务`，按页面业务对象命名。
- 表格行操作：`查看`、`编辑`、`复制`、`停用/作废/撤回`、`提交审批`、`查看履历`。
- 详情侧栏：基础信息、业务校验、下游影响、操作记录、关联单据。
- 表单弹窗：新增/编辑共用，必须包含责任人、来源、状态、生效范围、备注。
- 危险操作确认：删除、停用、作废、撤回、冻结、解冻必须二次确认，并写入审计记录。
- 数据持久化：新增、编辑、删除/停用、审批状态和筛选状态写入 localStorage。
- 空状态：无数据时提供新增入口；筛选无结果时提供清空筛选入口。
- 演示边界：凡是模拟扫码、PDA、PLC、WMS、ERP、工牌/NFC、设备 API 的按钮和区域必须明确出现“模拟”。
